Project Overview

La Tortuga EMR is a tablet-first medical and dental intake system built for La Tortuga Foundation's volunteer missions in rural Chiapas, Mexico. It captures patient data entirely offline, stores it locally on each tablet, and syncs automatically across the clinic network via a router-based mesh because in the field, there is no internet.

In March 2026, I led the deployment to Tonala, Chiapas across 8 Samsung Galaxy Tab Active3 tablets connected to a shared ASUS router. The system ran a full clinic workflow: patient check-in, medical and dental intake, a live waiting room queue, bilingual PDF report generation, and AI-powered community health summaries via a local Llama3 instance. It served 100+ patients and survived real-world conditions including a mid-deployment router failure and an emergency APK push.

System Architecture

The system is built on three independent layers. The clinic mesh handles all real-time operations completely offline. The cloud layer is optional and only used by a single designated tablet when internet is available. The reporting and AI layer runs fully offline on the local network.

Layer 1 — Clinic Mesh (Offline)

The primary operational layer runs within each clinic using a standard WiFi router. Tablets discover each other automatically via UDP broadcast on port 8888, then establish TCP connections on port 8889 for bidirectional delta sync. No special hardware is required.

Mesh Sync Flow

  1. Discovery: tablets broadcast UDP packets, peers detected automatically
  2. Connection: TCP channel opened on port 8889, bidirectional
  3. Delta Sync: only records modified since last sync are sent (24-hour window for reconnects)
  4. Conflict Resolution: higher syncVersion counter wins; services always merged field-by-field
  5. Confirmation: acknowledgment closes the cycle; timestamp updated for next sync

Identity & Conflict System

  • Azure-Peak IDs: each tablet generates a unique animal-mountain identifier (e.g. Blue-Elephant) at first launch
  • syncVersion counters: every record increments on update; highest wins on conflict
  • Field-level service merge: medical and dental completions always preserved, never overwritten
  • Auto sync: every 60 seconds; manual pull-to-refresh anytime (2-5 sec response)
Key design principle: each tablet holds a complete local copy of all clinic data. There is no single point of failure. If any tablet goes down, every other tablet continues operating independently.
Layer 2 — Cloud Analytics (Optional)

One designated Cloud Gateway tablet uploads de-identified aggregate data to Firebase / BigQuery when internet is available. All other tablets operate 100% offline. No PHI ever leaves a tablet.

Clinic Tablets (Router Mesh) | Cloud Gateway Tablet (1 designated tablet, when internet available) | Firebase / BigQuery | React / Vite Analytics Dashboard
Layer 3 — Offline Reporting & AI

The Report Center generates bilingual PDFs directly on-device. The AI Community Health feature aggregates and anonymizes patient data locally, then sends it to a local Ollama / Llama3 instance running on a laptop on the same clinic WiFi. No internet required at any step.

Clinic Tablets (Router Mesh) | Community Health Aggregator (anonymizes data locally, no PHI) | (WiFi only, no internet) Ollama / Llama3 (running on clinic laptop) | Bilingual AI Health Summary (ES + EN, displayed on tablet)

Field Challenges & Solutions

  • Router failure mid-deployment: tablets lost mesh connectivity on day 6. Each tablet's local SQLite data remained intact. Designed a post-trip identity-based SQL merge process to reconcile ~12 duplicate patient pairs.
  • UTC timezone rollover bug: patient records were dating to the wrong day at midnight local time. Patched with a shared dateUtils.ts local-time utility across all date-sensitive screens.
  • Partial-save crashes: an edge case in the multi-step intake flow caused incomplete records on app resume. Pushed emergency APK v2.1 mid-mission via ADB to all 8 tablets without disrupting clinic flow.
  • Community dropdown mismatches: tablet communities fell out of sync during propagation. Debugged data seeding logic in the field with targeted SQL queries.

Recent Improvements (February - March 2026)

  • Report Center: unified screen with mission calendar, single/multi-day selection, bilingual PDF preview and share
  • Community Health AI (beta): local Llama3 via Ollama, anonymized aggregation before any AI processing, configurable host IP
  • Edit Patient Screen: update records directly from Visit Hub, changes sync automatically
  • Router mesh migration: moved from WiFi Direct to standard router networking, simpler setup, better reliability
  • Delta sync + 24-hour window: only changed records sent, faster sync, prevents name corruption bug
