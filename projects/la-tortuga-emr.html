Project Overview

La Tortuga EMR is a production electronic medical records system I designed and built for La Tortuga Foundation's annual medical missions to underserved communities in southern Mexico. The system runs entirely offline across a local tablet network, syncing patient data peer-to-peer with no internet required — because in rural Chiapas, there is none.

In March 2026, I led the field deployment to Tonalá, Chiapas — 8 Samsung Galaxy Tab Active3 tablets, a local ASUS router, and a full clinic workflow covering patient check-in, medical intake, dental records, a live waiting room queue, and AI-generated community health reports. Over the course of the mission, the system served 100+ patients and survived real-world conditions including a mid-deployment router failure and an emergency APK push.

Role: Solo Developer (V1) & Tech Lead (V2 Capstone) • Timeline: Built in ~2.5 months, deployed March 2026 • Organization: La Tortuga Foundation

Key Features

  • Offline-first P2P sync across up to 8 tablets with no internet using cr-sqlite CRDT architecture.
  • Full patient workflow: check-in, medical intake, dental records, waiting room queue, and discharge.
  • AI-generated community health reports via local Ollama / Llama3 — bilingual English and Spanish output.
  • Firestore analytics dashboard (Vite / React) for de-identified aggregate data visualization.
  • Bilingual UI across all patient-facing screens with shared date utilities and JSON translation files.
  • Identity-based duplicate patient detection and post-mission SQL cleanup tooling.

Tech Stack

React Native + TypeScript + Expo
SQLite via op-sqlite + cr-sqlite (CRDT sync)
Firebase / Firestore (analytics pipeline)
Ollama / Llama3 (local AI, no internet required)
Vite + React (analytics dashboard)
ADB + Git (device management & version control)

Field Challenges & Solutions

  • Router failure mid-deployment: tablets lost sync connectivity on day 6. Resolved by isolating each tablet's local data and designing a post-trip SQL merge process to reconcile ~12 duplicate patient pairs.
  • UTC timezone bug: patient records were rolling over to the wrong date at midnight. Identified and patched with a shared local-time utility before deployment.
  • Partial-save crashes: an edge case in multi-step intake caused incomplete records. Pushed an emergency APK (V2.1) mid-mission via ADB to resolve without disrupting clinic flow.
  • Community dropdown mismatches: tablet communities fell out of sync during propagation. Debugged and corrected data seeding logic with targeted SQL queries in the field.

Outcome & Impact

  • 100+ patients served across a multi-day clinic in Tonalá, Chiapas with zero paper records.
  • All 8 tablets operated reliably in a zero-internet environment for the full mission duration.
  • AI-generated health reports delivered actionable community health insights to foundation leadership.
  • V2 in active development as an Aurora University capstone project, featuring a fully normalized 20-table schema and production-grade CRDT sync.
